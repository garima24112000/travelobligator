# Production Data Sources

## 1. Purpose

This document defines where TravelObligator gets real-world data from.

The goal is to make sure the system does not treat mock data, AI guesses, scraped data, incomplete provider responses, or unavailable provider data as production truth.

TravelObligator should separate:

* user input
* provider-backed facts
* open-data facts
* deterministic system calculations
* AI reasoning
* assumptions
* unavailable data
* user-provided but unverified external information

This document defines:

* which information comes from which source
* which sources are allowed for MVP
* which sources require future approval or partner access
* which data can be cached
* which data must be refreshed
* what AI is allowed to generate
* what AI must not invent
* what happens when providers fail
* how unavailable data should be handled
* how metasearch-style recommendations should work

---

## 2. Core Rule

Production data should never be silently replaced with fake data.

TravelObligator does not use mock data in MVP or production outputs.

If reliable data is unavailable, the system should say so.

Missing data should reduce confidence, not create hallucinated certainty.

The system must never use:

* mock accommodation listings
* mock flight results
* mock prices
* mock availability
* mock ratings
* mock review counts
* scraped provider data
* AI-generated factual provider data

If a data field cannot be verified through a legitimate source, the system must mark it as unavailable instead of filling it with mock, scraped, or AI-generated data.

---

## 3. Legit-Only Data Policy

All displayed facts must come from one of the following:

1. User-provided data
2. Official APIs
3. Open-data sources
4. Approved partner APIs
5. Approved affiliate APIs or feeds
6. Deterministic system calculations based on legitimate data
7. AI reasoning based only on the above

TravelObligator may use open data and official APIs for planning, routing, recommendations, and validation.

TravelObligator must not use restricted third-party website content unless the system has official API access, approved partner access, affiliate feed access, or another legally permitted integration.

If a source is not connected or not legally accessible, the system should mark that data as unavailable.

Unavailable data is acceptable. Fake data is not.

---

## 4. Data Source Categories

TravelObligator uses six categories of data.

### User-Provided Data

Comes directly from the traveler.

Examples:

* destination
* trip dates
* budget
* group size
* preferences
* constraints
* must-visit places
* must-avoid places
* free-text requirements
* user-selected accommodation
* user-selected flight option
* user-entered price seen on an external site

User-provided external information should be labeled clearly.

Example:

```json
{
  "price_per_night": 210,
  "data_status": "user_provided",
  "provider_verified": false,
  "message": "This price was entered by the user and should be rechecked on the provider site before booking."
}
```

### Provider Data

Comes from official APIs, approved APIs, approved affiliate feeds, or approved data providers.

Examples:

* places
* attractions
* restaurants
* ratings
* review counts
* coordinates
* opening hours
* routes
* travel times
* accommodation options
* accommodation prices
* accommodation availability when available
* flight prices when available
* flight availability when available

### Open Data

Comes from open-data sources that can be used under their license terms.

Examples:

* OpenStreetMap / Overpass
* GTFS transit feeds
* OpenTripPlanner routing outputs
* Open-Meteo weather
* Nager.Date public holidays
* Frankfurter currency conversion
* GeoNames location metadata

Open data must still be handled with attribution, license compliance, usage limits, and source labeling.

### System-Computed Data

Computed deterministically by TravelObligator.

Examples:

* trip duration
* daily activity count limits
* budget comparisons
* walking thresholds
* route grouping
* attraction clustering
* accommodation scoring
* validation severity
* confidence scoring
* version history

### AI-Generated Reasoning

Generated by the LLM from user input, provider data, open data, system-computed values, and visible assumptions.

Examples:

* preference interpretation
* trip style explanation
* decision card wording
* subjective quality reasoning
* feedback interpretation
* change explanations
* ranking explanations

AI reasoning must never become the factual source for provider-backed fields.

### Assumptions

Used when information is missing but the system can safely continue.

Examples:

* assume moderate meal duration
* assume hotel budget is part of total trip budget
* assume no accessibility needs if user did not mention any
* assume standard walking comfort unless the user gives mobility constraints

Assumptions must be visible in metadata or explanation cards.

## 5. MVP Legit Data Sources

TravelObligator MVP should use only legitimate data sources.

Available for MVP:

* OpenStreetMap / Overpass for places, attractions, restaurants, and accommodation POIs
* Nominatim or GeoNames for destination and location resolution where suitable
* OpenTripPlanner + GTFS + OpenStreetMap for open-source routing and transit planning
* Google Places API as an optional provider for richer place metadata
* Google Routes API or Mapbox Directions as optional routing providers
* Open-Meteo for weather
* Nager.Date for public holidays
* Frankfurter for currency conversion
* Amadeus Flight Offers Search for flight search where production access is available
* Amadeus Hotel APIs for hotel-focused accommodation inventory where production access is available
* OpenAI Structured Outputs for reasoning, explanation, validation, and feedback interpretation

Unavailable unless approved provider access is connected:

* Airbnb listings, prices, calendars, and availability
* Booking.com live accommodation inventory
* Expedia and Vrbo live vacation-rental inventory
* Google Flights backend search results
* Tripadvisor pricing or restricted content
* final booking confirmation for hotels, rentals, tours, or flights

If a data source is not connected or legally accessible, the system must mark that data as unavailable and must not replace it with mock data, scraped data, or AI-generated facts.

## 7. Recommended MVP Providers

### Places Provider

Primary:

* OpenStreetMap / Overpass
* Google Places API when richer metadata is needed and API usage is acceptable

Used for:

* attractions
* restaurants
* cafes
* landmarks
* museums
* parks
* viewpoints
* shopping areas
* coordinates
* categories
* opening hours when available
* ratings and review counts only when returned by an approved provider

Fallback:

* OpenStreetMap / Overpass where suitable
* Foursquare or another approved places provider where available

Rules:

* Do not invent places.
* Do not invent ratings.
* Do not invent opening hours.
* Do not invent review counts.
* If provider metadata is missing, mark it as unavailable.
* If using OpenStreetMap only, do not imply that ratings, review counts, price levels, or live opening status were verified.

---

### Routes Provider

Primary:

* OpenTripPlanner + OpenStreetMap + GTFS for open-source routing and transit feasibility
* Google Routes API when official API usage is available

Fallback:

* Mapbox Directions API

Used for:

* walking routes
* driving routes
* transit feasibility when available
* scheduled transit routing
* travel time
* distance
* route comparison
* route matrix calculations
* movement feasibility validation

Rules:

* AI must not invent exact travel times.
* AI must not invent walking distances.
* If route data is unavailable, the itinerary should be marked lower confidence.
* If missing route data affects feasibility, the validator should flag it.
* If live transit data is unavailable, transit estimates should be labeled as scheduled, cached, or unavailable.

---

### Transit Data Policy

Transit feasibility should be provider-backed or open-data-backed.

Allowed sources:

* GTFS feeds
* OpenTripPlanner
* Transitland feed discovery where suitable
* Google Routes API where available
* Mapbox where suitable

Transit outputs should distinguish between:

* live transit
* scheduled transit
* cached transit
* unavailable transit

The system must not invent transit feasibility, route duration, walking transfer time, or stop information.

---

### Accommodation Provider

Primary:

* OpenStreetMap / Overpass for real accommodation POI discovery
* Amadeus Hotel APIs for hotel-focused price and availability where production access is available
* Approved broad lodging provider where available

The provider should support multiple accommodation types where possible, including:

* hotels
* motels
* hostels
* resorts
* serviced apartments
* guesthouses
* boutique stays
* vacation rentals

If the selected provider only supports hotel-like inventory, the system must clearly label accommodation coverage as limited.

Possible provider categories:

* open accommodation POI provider
* hotel provider
* hostel provider
* vacation rental provider
* resort provider
* serviced apartment provider
* aggregator provider
* metasearch or affiliate provider

Examples of possible approved sources:

* OpenStreetMap / Overpass for real accommodation POIs
* Amadeus Hotel APIs for hotel-focused inventory
* Booking.com Demand API with approved affiliate access
* Expedia Rapid with approved partner access
* Vrbo through Expedia Rapid with approved partner access
* Hostelworld Partner API/feed with approved access
* Hotelbeds with approved access
* Skyscanner Hotels API with approved access
* Travelpayouts or similar affiliate API/feed where allowed

Used for:

* accommodation names
* accommodation type
* area
* coordinates
* website when available
* nightly price when available
* availability when available
* rating when available
* review count when available
* amenities when available
* provider URL or deeplink when available
* cancellation information when available

### Accommodation Coverage Matrix

| Accommodation Type  | Allowed Source                                                                                      |
| ------------------- | --------------------------------------------------------------------------------------------------- |
| Hotels              | OpenStreetMap / Overpass, Amadeus Hotels, Booking.com Demand API, Expedia Rapid, Hotelbeds          |
| Motels              | OpenStreetMap / Overpass, Booking.com Demand API, Expedia Rapid, Hotelbeds                          |

* If only OpenStreetMap accommodation POIs are available, the system may show real accommodation locations but must mark price, availability, rating, and review count as unavailable unless present from a legitimate provider.

### Flight Provider Policy

Flights are not required for the core single-city MVP unless the product includes trip cost estimation, arrival planning, departure planning, or transportation comparison.

If flights are included, use approved flight APIs such as:

* Amadeus Flight Offers Search
* Duffel
* approved airline APIs
* approved GDS provider
* approved metasearch or affiliate feed

Google Flights may be used only as a user-facing reference or external provider handoff if allowed by the integration terms.

Google Flights should not be treated as a normal public backend API.

The system must not:

* scrape Google Flights
* scrape unsupported OTA pages
* invent fares
* invent baggage rules
* invent seat availability
* invent airline schedules
* invent booking links
* show test-environment data as live production truth

Flight outputs should include:

* airline
* origin
* destination
* departure time
* arrival time
* price when provider-confirmed
* availability when provider-confirmed
* baggage details when provider-confirmed
* provider/source
* freshness
* confidence
* external handoff link when legally provided by the source

---

### Metasearch / Referral Model

TravelObligator does not complete bookings inside the MVP.

The system may recommend provider-backed options and redirect users to the original provider, airline, hotel, OTA, or booking platform for final booking.

This is allowed only when the option, price, availability, and redirect link come from:

* an official API
* an approved partner API
* an affiliate API/feed
* an approved provider deeplink
* open data where no price or availability is claimed
* user-provided information clearly labeled as not provider-verified

The system may show:

* provider-backed flight options
* provider-backed accommodation options
* provider-backed prices
* provider-backed availability
* provider name
* source freshness
* confidence
* “View deal” or “Continue on provider site” button

The system must not:

* scrape Booking.com, Airbnb, Expedia, Vrbo, Tripadvisor, Google Flights, or similar sites
* use browser automation to collect provider results
* display copied prices, ratings, reviews, photos, calendars, or availability without approved data access
* imply that a restricted provider was searched when no approved integration exists

Final booking happens outside TravelObligator.

TravelObligator’s role is recommendation, comparison, validation, and provider handoff.

---

### External Platform Access Rule

TravelObligator must not automatically search, scrape, extract, copy, or redisplay results from restricted third-party platforms such as:

* Airbnb
* Booking.com
* Expedia
* Vrbo
* Google Flights
* Tripadvisor
* similar OTA, travel, map, review, or booking platforms

unless official API, affiliate, or partner access is available.

The system may:

* direct the user to check an external platform manually
* accept user-entered accommodation or flight details as user-provided data
* label user-entered provider information as not provider-verified
* use official APIs or approved partner integrations when available
* redirect users through approved deeplinks when allowed

The system must not:

* use browser automation to collect third-party listings
* scrape search result pages
* copy prices, ratings, reviews, photos, calendars, or availability into the dashboard
* imply that restricted platforms were searched when no approved integration exists

## 11. Cache Rules

Caching is allowed only when it does not mislead the user and only when the provider/license permits it.

### Cacheable

* place coordinates
* place categories
* attraction metadata
* neighborhood candidates
* destination context
* route matrix for non-live estimates when allowed
* provider-backed restaurant details when allowed
* provider-backed accommodation metadata when allowed
* OpenStreetMap-based POI metadata when license obligations are followed
* public holiday data
* currency conversion results for short periods

### Short Cache Only

* accommodation price
* accommodation availability
* flight price
* flight availability
* route travel time
* restaurant opening hours
* weather
* live transit estimates

### Should Not Be Reused Across Users Without Care

* Traveler Profile
* Trip Strategy
* Stay + Transport decisions
* Experience Plan
* Validation Report
* Feedback Interpretation
* user-entered accommodation or flight options

These are user-specific and should be stored as part of the user's Planning State, not reused globally.

## 14. Provider Failure Policy

When a provider fails, the system should not continue as if reliable data was available.

The system should:

1. Retry the provider call when appropriate.
2. If retry fails, use a fallback provider when available.
3. If fallback data is used, label it clearly.
4. If no reliable data is available, mark the field as unavailable or low confidence.
5. Never silently replace failed provider data with fake data, mock data, scraped data, or AI-generated facts.

Provider failure should reduce confidence, not create hallucinated certainty.

Example:

```json
{
  "travel_time_minutes": null,
  "data_status": "unavailable",
  "provider_status": "failed",
  "confidence": 0.0,
  "user_message": "Travel time could not be verified from the routing provider."
}
```

Fallback example:

```json
{
  "travel_time_minutes": 25,
  "data_status": "fallback_used",
  "data_source": "mapbox_directions",
  "provider_status": "fallback_used",
  "confidence": 0.72
}
```

Provider-not-connected example:

```json
{
  "source": "Booking.com",
  "data_status": "not_connected",
  "provider_status": "not_connected",
  "confidence": 0.0,
  "user_message": "Booking.com inventory was not searched because no approved provider integration is connected."
}
```

## 16. Safety Data Policy

The MVP does not generate direct safety ratings for:

* destinations
* neighborhoods
* routes
* hotels
* accommodation options
* restaurants
* attractions

Instead, the system identifies safety-related planning considerations such as:

* late-night travel
* long walking segments
* poor transit alignment
* remote or isolated movement
* weather exposure
* traveler-specific comfort constraints
* long transfers
* route uncertainty
* low provider confidence

The system may suggest lower-friction alternatives, but it should not label a place as safe or unsafe without authoritative data.

The system should say:

```text
This option may require late-night travel and a long walk from transit.
```

The system should not say:

```text
This area is unsafe.
```

unless supported by a specific authoritative source and the product is designed to handle that responsibly.

## 17. Accommodation Data Policy

The MVP recommends accommodation options, not final bookings.

Accommodation options may include:

* hotels
* motels
* hostels
* resorts
* serviced apartments
* vacation rentals
* guesthouses
* boutique stays

only when supported by the connected data source.

The system should recommend the best available accommodation options according to:

* selected stay area
* accommodation type suitability
* budget fit when price is available
* review score when available
* review count when available
* distance to planned attraction clusters
* transport access
* amenities when available
* parking needs when available
* comfort requirements
* traveler group type
* data freshness
* provider confidence
* provider coverage

The system should not guarantee:

* final price
* final availability
* booking completion
* cancellation terms
* platform coverage
* all accommodation types searched

unless confirmed by a provider.

If only hotel data is available from the current provider, the system should say:

```text
Accommodation results are limited to hotel-like inventory from the connected provider.
```

If only OpenStreetMap data is available, the system should say:

```text
Accommodation results are based on real accommodation locations from open data. Live prices, availability, ratings, and review counts are unavailable because no approved booking provider is connected.
```

If vacation rentals or Airbnb-style inventory are unavailable, the system should say:

```text
Vacation-rental and Airbnb-style inventory are unavailable because no approved provider integration is connected.
```

* distance or travel time when available
* rating when available
* review count when available

* opening hours when available
* why it fits the traveler
* data source
* confidence

The system should not invent:

* restaurant names
* ratings
* review counts
* opening hours
* prices
* availability

If using OpenStreetMap only, the system may show real restaurant/cafe names and locations, but should not imply that ratings, price level, or review counts were checked.

If reliable restaurant data is unavailable, the system should recommend meal areas instead of fake restaurant names.

## 22. MVP Implementation Order

Recommended implementation order for a legit, no-mock MVP:

1. OpenAI Structured Outputs for structured reasoning, validation, and feedback interpretation.
2. OpenStreetMap / Overpass for attractions, restaurants, accommodation POIs, and place discovery.
3. Nominatim or GeoNames for destination and location resolution.
4. OpenTripPlanner + GTFS + OpenStreetMap for routing and transit feasibility where available.
5. Open-Meteo for weather-aware planning.
6. Nager.Date for public holidays.
7. Frankfurter for currency conversion.
8. Amadeus Flight Offers Search for flight options where production access is available.
9. Amadeus Hotel APIs for hotel-focused prices and availability where production access is available.
10. Google Places API as an optional richer place provider.
11. Google Routes API or Mapbox Directions as optional routing providers.
12. Approved partner or affiliate providers for broader accommodation and metasearch coverage when available.

The MVP should not depend on Booking.com, Expedia, Vrbo, Airbnb, Tripadvisor, or Google Flights backend data unless approved provider access exists.

## 23. MVP Scope Statement

TravelObligator MVP uses real open/provider-backed data for places, routes, weather, holidays, currency, flights, and hotel-focused accommodation where available.

Vacation rentals, Airbnb, Booking.com, Expedia, Vrbo, Tripadvisor pricing, and Google Flights backend results are marked unavailable unless official provider access is connected.

The MVP goal is:

```text
complete itinerary planning and decision support in one platform
```

not:

```text
replicating restricted booking marketplaces
```

TravelObligator may support a metasearch/referral model, where provider-backed options are shown in the dashboard and final booking happens on the original provider website.

## 24. Design Principles

Production data sources should follow these principles:

* Use real provider data where possible.
* Use open data where allowed and useful.
* Never use mock data in MVP or production outputs.
* Never treat mock data as production truth.
* Never invent provider-backed facts.
* Never scrape unsupported sources.
* Never imply unavailable providers were searched.
* Label fallback data clearly.
* Label unavailable data clearly.
* Label user-provided unverified data clearly.
* Separate facts from reasoning.
* Separate estimates from facts.
* Separate provider-backed data from open data.
* Separate provider-confirmed data from user-provided data.
* Cache only when safe and allowed.
* Reduce confidence when data is missing.
* Store provider status in Planning State.
* Store provider coverage in Planning State.
* Keep the provider layer replaceable.
* Use metasearch/referral only through approved APIs, feeds, or deeplinks.
* The user should know when the system is confident and when it is making assumptions.
* The user should know what data sources were searched and what sources were unavailable.
